# Key Code Fixes

This document describes the five concrete code changes that moved needle retrieval from 0% to 100% at 16K tokens. No patch files were extracted from the source repositories, so each fix is described in terms of the file affected, the error, and the correction applied.

## Fix 1: QJL Orthogonal Projection (mlx-optiq / turboquant_plus)

**File:** `optiq/core/turbo_quant.py` (installed mlx-optiq package) and `turboquant_plus/turboquant/qjl.py`

**Error:** The QJL stage used a random Gaussian matrix S for the projection `sign(S * residual)`. Gaussian projections are unbiased in expectation but introduce variance of order `||r||^2 / d` per dimension. For real LLM KV vectors with head dimension 128, this variance is large relative to the signal. Every attention step accumulates a small error, and over thousands of tokens the attention scores collapse. The symptom was word-loop degeneration ("genetic genetic genetic...") the moment QJL was enabled.

**Fix:** Replace the Gaussian matrix with a random orthogonal matrix generated by QR decomposition of a Gaussian draw. Orthogonal matrices preserve norms and inner products. The variance of the reconstruction drops sharply.

```python
# Replacement for QJL matrix initialization
import mlx.core as mx

def generate_orthogonal_matrix(d, seed=42):
    key = mx.random.key(seed)
    G = mx.random.normal(shape=(d, d), key=key)
    Q, R = mx.linalg.qr(G, stream=mx.cpu)
    mx.eval(Q, R)
    diag_signs = mx.sign(mx.diag(R))
    Q = Q * diag_signs[None, :]
    mx.eval(Q)
    return Q
```

## Fix 2: QJL Dequantization Scale Factor (mlx-optiq / turboquant_plus)

**File:** `optiq/core/turbo_quant.py` (installed mlx-optiq package)

**Error:** The dequantization formula applied a scale of `sqrt(pi/2) / d` to the QJL correction term. For an orthogonal basis in d dimensions, the correct unbiased estimator requires `sqrt(pi/2) / sqrt(d)`. At d=128, the incorrect formula is approximately 11 times too small (sqrt(128) = 11.31). This effectively disabled the QJL correction even when it was switched on, because the correction term was negligible.

**Fix:** Change the scale to `sqrt(pi/2) / sqrt(d)`.

```python
# Incorrect (original):
scale = math.sqrt(math.pi / 2.0) / self.d

# Correct:
scale = math.sqrt(math.pi / 2.0) / math.sqrt(self.d)
```

This single character change allowed the QJL stage to reduce quantization MSE by the theoretical 64 percent for the first time.

**Source:** `reports/round1-post-mortem-report.md`, `benchmarks/test_hybrid_needle.py`

## Fix 3: GGML Context Sizing Bug (TheTom llama-cpp-turboquant fork)

**File:** `src/llama-kv-cache.cpp`, line 54

**Error:** The GGML metadata context pre-allocates space for tensor descriptor structs. The original formula counted exactly `2 * (1 + n_stream) * n_layer_kv` slots, one for each K and V tensor per layer. But TurboQuant types allocate two additional shared tensors (turbo_rotation and turbo_rotation_inv, each 128x128 float32) after the layer loop. These were not counted. When they attempted to allocate inside the already-full context, `ggml_new_tensor_impl` returned null and `GGML_ASSERT(obj_new)` failed. The crash happened even with `-ngl 0` (GPU disabled), so the initial hypothesis that this was a Metal Tensor API incompatibility was wrong.

**Fix:** Add space for two additional tensor descriptors.

```cpp
// Incorrect (original):
/*.mem_size =*/ size_t(2u*(1 + n_stream)*n_layer_kv*ggml_tensor_overhead()),

// Correct:
/*.mem_size =*/ size_t(2u*(1 + n_stream)*n_layer_kv*ggml_tensor_overhead()) + 2*ggml_tensor_overhead(),
```

**Source:** `reports/round1-experiment-log.md` Phase 4 section

## Fix 4: Norm Correction in tq3_0 Quantizer (Aaryan Kapoor llama.cpp fork)

**Files:**
* `ggml/src/ggml-quants.c`
* `ggml/src/ggml-metal/ggml-metal.metal`

**Error:** The tq3_0 quantizer stored the raw RMS of the input block in `block_tq3_0.d`. After Lloyd-Max quantization and inverse Walsh-Hadamard reconstruction, the decoded block norm no longer matched the original block norm. The difference between original and reconstructed norm corrupted key vector magnitude and damaged query-key dot products. This was the primary cause of K-path correctness failures (repetitive degenerate text from `K=tq3_0, V=f16` runs while `K=f16, V=tq3_0` remained coherent).

**Fix:** Quantize and reconstruct first, then measure the reconstructed norm, then store `original_norm / reconstruction_norm` as the scale factor. On decode, multiply by this factor to recover the correct magnitude.

Pseudocode for the corrected quantizer:
```
1. Measure original_norm = ||x||
2. Normalize: x_hat = x / original_norm
3. Rotate via WHT
4. Quantize to Lloyd-Max centroids, storing indices
5. Reconstruct: x_recon = dequantize(indices) via inverse WHT
6. Measure reconstruction_norm = ||x_recon||
7. Store d = original_norm / reconstruction_norm
```

## Fix 5: Zero Block Handling in tq3_0 Quantizer (Aaryan Kapoor llama.cpp fork)

**Files:**
* `ggml/src/ggml-quants.c`
* `ggml/src/ggml-metal/ggml-metal.metal`

**Error:** When an input block had near-zero energy, the original code forced `rms = 1.0` as a guard against division by zero. This caused zero-energy blocks to decode as structured nonzero values, injecting noise into the attention mechanism on every token where that block would legitimately be zero.

**Fix:** Detect near-zero blocks and emit a true zero block: set `d = 0` and fill all packed quantization data with zeros. On decode, a zero scale factor produces a zero output block with no noise.

```c
// Corrected zero-block guard:
if (original_norm < 1e-9f) {
    block->d = 0;
    memset(block->qs, 0, sizeof(block->qs));
    continue;
}
```
